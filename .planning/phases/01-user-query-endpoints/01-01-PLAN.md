**UserQueryRepository.java** — port interface in `org.athlium.users.domain.repository`:
- This is a NEW interface separate from the existing `UserRepository`. It defines the cross-module query operations.
- Methods:
  1. `PageResponse<UserWithPackageStatus> findUsersByHeadquarters(Long headquartersId, String status, String search, int page, int size, String sort)` — page is 0-indexed
  2. `PageResponse<UserWithPackageStatus> findUsersByOrganization(Long organizationId, String status, String search, int page, int size, String sort)` — returns flat list; org-level HQ grouping is a separate method
  3. `List<UserHqMembership> findHqMembershipsByUserIds(List<Long> userIds, Long organizationId)` — fetches HQ memberships for a list of user IDs within an org (used by GetUsersByOrg to enrich paginated results)
  4. `PageResponse<UserWithPackageStatus> findAllUsers(String status, String search, int page, int size, String sort)` — all users with best-overall status
  5. `UserWithPackageStatus findUserById(Long userId)` — single user with best-overall status; returns null if not found
- Import only from `org.athlium.users.domain.model.*`, `org.athlium.shared.domain.PageResponse`, `java.util.List`

**GetUsersByHqUseCase.java** — `@ApplicationScoped` use case:
- `@Inject UserQueryRepository userQueryRepository` (package-private field injection)
- `public PageResponse<UserWithPackageStatus> execute(Long headquartersId, String status, String search, int page, int size, String sort)`
- Validation: headquartersId must not be null (throw `BadRequestException`), page >= 1 (throw `BadRequestException("Page must be >= 1")`), size between 1-100 (throw `BadRequestException("Size must be between 1 and 100")`)
- If status is provided, validate it's a valid `PackageStatus` string (use `PackageStatus.fromString()`)
- Convert page from 1-indexed (API) to 0-indexed (repository): `page - 1`
- Default sort to "name:asc" if null/empty
- Delegate to `userQueryRepository.findUsersByHeadquarters()`

**GetUsersByOrgUseCase.java** — `@ApplicationScoped` use case:
- Same validation pattern as GetUsersByHq but with `organizationId`
- This use case has extra logic: after getting the paginated `PageResponse<UserWithPackageStatus>` from the repository, it needs to enrich each user with their HQ memberships within the org
- Steps:
  1. Call `userQueryRepository.findUsersByOrganization()` to get paginated users
  2. Extract user IDs from the page results
  3. Call `userQueryRepository.findHqMembershipsByUserIds(userIds, organizationId)` to get HQ details
  4. Group memberships by user ID and attach to each user (set on `UserWithPackageStatus` — need a `List<UserHqMembership> hqMemberships` field on it, or create a new domain model `UserWithOrgDetails`)
- IMPORTANT DESIGN DECISION: Add a `List<UserHqMembership> hqMemberships` field to `UserWithPackageStatus` (nullable — only populated for org queries). This avoids creating a separate domain model. When hqMemberships is populated, the top-level packageStatus represents the best status across all HQs in the org.
- Return type: `PageResponse<UserWithPackageStatus>` (same as others, but hqMemberships field populated)

**GetAllUsersUseCase.java** — `@ApplicationScoped` use case:
- Same validation as GetUsersByHq (page, size, status, search, sort) but no headquartersId/organizationId required
- Delegate to `userQueryRepository.findAllUsers()`
- This endpoint returns ALL users (including those with NO_PACKAGE — those who never bought a package)

**GetUserByIdUseCase.java** — `@ApplicationScoped` use case:
- `@Inject UserQueryRepository userQueryRepository`
- `@Inject UserRepository userRepository` (existing — for fallback if user has no packages)
- `public UserWithPackageStatus execute(Long userId)`
- Validation: userId must not be null
- Call `userQueryRepository.findUserById(userId)` — returns user with best-overall package status
- If null, check if user exists via `userRepository.findById(userId)` — if exists, return a UserWithPackageStatus with NO_PACKAGE status. If not exists, throw `EntityNotFoundException("User not found")`
- This ensures users without any packages still return a valid response

All use cases follow existing pattern: `@ApplicationScoped`, package-private `@Inject` fields, single `execute()` method, `@Transactional` NOT needed (read-only queries).
