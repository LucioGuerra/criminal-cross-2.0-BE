  <action>
Create `UserQueryRepositoryImpl` in `org.athlium.users.infrastructure.repository` that implements `UserQueryRepository`. This is the most complex file in the phase — it executes native SQL queries across the users → client_packages → client_package_credits → activity → headquarters table chain.

**Class structure:**
- `@ApplicationScoped`
- `@Inject EntityManager em` (jakarta.persistence.EntityManager — for native SQL)
- Implements all 5 methods from `UserQueryRepository`

**SQL approach per CONTEXT.md locked decisions:**
- Package status computation: SQL CASE WHEN (not application-level)
- Data resolution: Single SQL query with JOINs
- Best status wins: Use CTE with ROW_NUMBER() window function

**Method 1: `findUsersByHeadquarters(Long headquartersId, String status, String search, int page, int size, String sort)`**
- Base SQL uses CTE pattern from RESEARCH.md:
  1. CTE `user_packages`: JOIN users → client_packages → client_package_credits → activity, filter by `a.hq_id = :hqId`, compute per-package `status_rank` (1=ACTIVE, 2=EXPIRING, 3=INACTIVE) and `package_status` string using CASE WHEN:
     - `cp.active = false` → INACTIVE (revoked)
     - `cp.period_end < CURRENT_DATE` → INACTIVE (expired)
     - `cp.period_end <= CURRENT_DATE + 3` → EXPIRING
     - else → ACTIVE
  2. CTE `best_status`: ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY status_rank ASC, period_end DESC) to pick best package per user
  3. SELECT from best_status WHERE rn = 1
- Dynamic WHERE clauses: if `status` param provided, add `AND package_status = :status`; if `search` param provided, add `AND (LOWER(name) LIKE :search OR LOWER(last_name) LIKE :search OR LOWER(email) LIKE :search)` with `search = "%" + search.toLowerCase() + "%"`
- Dynamic ORDER BY: parse `sort` param (format: "field:direction", default "name:asc"). Allowed fields: `name`, `last_name`, `email`. Map to SQL column names. Reject invalid fields.
- Pagination: `LIMIT :size OFFSET :offset` where offset = page * size
- Count query: same CTEs but `SELECT COUNT(*)` instead of selecting rows — use `COUNT(DISTINCT u.id)` from the base JOIN (before CTE dedup), with same status/search filters applied after dedup
- Map results: `List<Object[]>` rows → `UserWithPackageStatus` domain objects via a private `mapRow(Object[] row)` method
- Roles: After getting paginated user IDs, execute a SECOND query: `SELECT ur.user_id, ur.role FROM user_roles ur WHERE ur.user_id IN (:userIds)` — group by user_id and set roles on each `UserWithPackageStatus`

**Method 2: `findUsersByOrganization(Long organizationId, ...)`**
- Same pattern as findUsersByHeadquarters but:
  - JOIN extends to headquarters: `JOIN headquarters h ON h.id = a.hq_id WHERE h.organization_id = :orgId`
  - The "best status" is across all HQs in the org (not per HQ — that's the hqMemberships method's job)
  - Everything else (status filter, search, sort, pagination, roles) identical

**Method 3: `findHqMembershipsByUserIds(List<Long> userIds, Long organizationId)`**
- Query: For each user in the list, find their HQ memberships within the org
- SQL: JOIN users → client_packages → client_package_credits → activity → headquarters, filter by `h.organization_id = :orgId AND u.id IN (:userIds)`, use same CTE best-status pattern but PARTITION BY (user_id, hq_id) to get best status per user-HQ combination
- Returns `List<UserHqMembership>` with hqId, hqName, packageStatus, periodEnd, daysRemaining
- If userIds is empty, return empty list immediately (avoid SQL IN with empty list)

**Method 4: `findAllUsers(String status, String search, int page, int size, String sort)`**
- CRITICAL: Use LEFT JOIN (not INNER JOIN) from users to the package chain so users with NO packages still appear
- For users with no matching packages, package_status = 'NO_PACKAGE', period_end = NULL, days_remaining = NULL
- Same best-status CTE pattern, but with COALESCE for NULL handling: `COALESCE(package_status, 'NO_PACKAGE')` for users with no packages
- If status filter = 'NO_PACKAGE', filter for users where the LEFT JOIN produced NULLs (no packages at all)
- Pagination, search, sort, roles query — same as other methods

**Method 5: `findUserById(Long userId)`**
- Single user query: LEFT JOIN from users to package chain, filter by `u.id = :userId`
- Use same best-status CTE but for one user
- Return null if user doesn't exist (let use case handle the 404)
- Roles: single query for this user's roles

**Private helper methods:**
- `mapRow(Object[] row)` → `UserWithPackageStatus`
- `mapHqMembershipRow(Object[] row)` → `UserHqMembership`
- `fetchRolesForUsers(List<Long> userIds)` → `Map<Long, Set<Role>>`
- `buildSortClause(String sort)` → validated SQL ORDER BY fragment (prevent SQL injection by allowlisting field names)
- `addSearchFilter(StringBuilder sql, Map<String, Object> params, String search)` — appends LIKE conditions
- `addStatusFilter(StringBuilder sql, Map<String, Object> params, String status)` — appends status condition

**SQL injection prevention:** NEVER concatenate user input into SQL strings. Use named parameters (`:paramName`) for all dynamic values. For sort field names, use an allowlist map (e.g., `Map.of("name", "name", "lastName", "last_name", "email", "email")`).

- Plain POJO with explicit getters and setters

**UserQueryDtoMapper.java** — MANUAL mapper (not MapStruct) in `org.athlium.users.presentation.mapper`:
- `@ApplicationScoped` CDI bean (manual because the mapping involves enum→string conversion and conditional HQ memberships)
- Methods:
  - `UserWithStatusResponse toResponse(UserWithPackageStatus domain)` — maps all fields, converts `PackageStatus` enum to string via `.name()`, converts `Set<Role>` to `Set<String>` via stream map, maps hqMemberships if non-null
  - `UserHqMembershipResponse toHqMembershipResponse(UserHqMembership domain)` — maps HQ membership fields
  - `List<UserWithStatusResponse> toResponseList(List<UserWithPackageStatus> domains)` — convenience list mapper

**UserQueryResource.java** — REST controller in `org.athlium.users.presentation.controller`:
- `@Path("/api/users")` — same base path as existing `UserResource` (JAX-RS supports multiple classes on same path)
- `@Produces(MediaType.APPLICATION_JSON)` and `@Consumes(MediaType.APPLICATION_JSON)`
- `@Authenticated(roles = {"SUPERADMIN", "ORG_ADMIN"})` at class level (per AUTH-01)
- Inject all 4 use cases + `UserQueryDtoMapper`

- **GET /api/users** (method: `getUsers`):
  - Query params: `@QueryParam("headquartersId") Long headquartersId`, `@QueryParam("organizationId") Long organizationId`, `@QueryParam("status") String status`, `@QueryParam("search") String search`, `@DefaultValue("1") @QueryParam("page") int page`, `@DefaultValue("20") @QueryParam("size") int size`, `@DefaultValue("name:asc") @QueryParam("sort") String sort`
  - Routing logic:
    - If `headquartersId != null` → call `getUsersByHqUseCase.execute(headquartersId, status, search, page, size, sort)`
    - Else if `organizationId != null` → call `getUsersByOrgUseCase.execute(organizationId, status, search, page, size, sort)`
    - Else → call `getAllUsersUseCase.execute(status, search, page, size, sort)`
  - If both headquartersId and organizationId provided → return 400 with `ApiResponse.error("Cannot filter by both headquartersId and organizationId")`
  - Map `PageResponse<UserWithPackageStatus>` → create a new `PageResponse<UserWithStatusResponse>` using `userQueryDtoMapper.toResponseList()` on the content
  - Return `Response.ok(ApiResponse.success("Users retrieved", mappedPageResponse)).build()`
  - Error handling: try-catch for `BadRequestException` → 400, `EntityNotFoundException` → 404, `Exception` → 500

- **GET /api/users/{id}** (method: `getUserById`):
  - `@PathParam("id") Long id`
  - Call `getUserByIdUseCase.execute(id)`
  - Map to response DTO
  - Return `Response.ok(ApiResponse.success("User found", response)).build()`
  - Error handling: `EntityNotFoundException` → 404, `BadRequestException` → 400, `Exception` → 500

NOTE: The existing `UserResource` already has `GET /api/users/firebase/{uid}`. Our `GET /api/users/{id}` uses a Long path param which won't conflict because "firebase" is a string literal sub-path, not a path param. JAX-RS resolves literal paths before parameterized ones.
