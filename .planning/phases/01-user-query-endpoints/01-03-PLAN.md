Create four use case test classes following the project's established test patterns (see TESTING.md and reference `CreateBookingUseCaseTest.java`). No Mockito — use hand-written inner static class test doubles only.

**Shared test infrastructure (inner static class in each test file):**

`InMemoryUserQueryRepository` — implements `UserQueryRepository`:
- Store a list of `UserWithPackageStatus` and a list of `UserHqMembership` as internal state
- Each method returns pre-configured data filtered by the parameters passed (enough to verify the use case calls the right method with the right args)
- Capture the parameters passed to each method for assertion (e.g., `lastHeadquartersId`, `lastPage`, `lastSize`)
- Helper methods: `addUser(UserWithPackageStatus)`, `addHqMembership(Long userId, UserHqMembership)`, `setFindUserByIdResult(UserWithPackageStatus)`

**GetUsersByHqUseCaseTest.java:**
- `@BeforeEach`: create `GetUsersByHqUseCase`, assign `InMemoryUserQueryRepository` to `useCase.userQueryRepository`
- Tests:
  1. `shouldReturnUsersForValidInput` — call with headquartersId=1L, page=1, size=20 — assert repository received page=0 (1-indexed to 0-indexed conversion), assert result matches repository output
  2. `shouldThrowWhenHeadquartersIdIsNull` — `assertThrows(BadRequestException.class, () -> useCase.execute(null, ...))`
  3. `shouldThrowWhenPageIsLessThanOne` — page=0 → BadRequestException
  4. `shouldThrowWhenSizeIsOutOfRange` — size=0 and size=101 → BadRequestException
  5. `shouldThrowWhenStatusIsInvalid` — status="INVALID" → BadRequestException (from PackageStatus.fromString)
  6. `shouldAcceptValidStatusFilter` — status="ACTIVE" → no exception, result returned
  7. `shouldDefaultSortToNameAsc` — sort=null → repository receives "name:asc"
- Helper: `createTestUser(Long id, String name, PackageStatus status)` → creates a `UserWithPackageStatus` with test data

**GetUsersByOrgUseCaseTest.java:**
- Same validation tests as GetUsersByHq (null orgId, page, size, status, sort)
- Additional tests:
  1. `shouldEnrichUsersWithHqMemberships` — configure repository to return 2 users and HQ memberships for each → assert each user in result has `hqMemberships` populated
  2. `shouldHandleEmptyUserListWithoutCallingHqMemberships` — repository returns empty page → verify HQ memberships query not called (use parameter capture or a flag in the in-memory repo)
  3. `shouldReturnEmptyHqMembershipsForUsersWithNone` — repository returns users but no HQ memberships → assert hqMemberships is empty list (not null)

**GetAllUsersUseCaseTest.java:**
- Same validation tests for page, size, status, sort (NO headquartersId/organizationId)
- Tests:
  1. `shouldReturnAllUsersForValidInput` — assert delegates to `findAllUsers` method
  2. `shouldPassStatusFilterToRepository` — status="NO_PACKAGE" → repository receives it
  3. `shouldPassSearchToRepository` — search="john" → repository receives it

**GetUserByIdUseCaseTest.java:**
- `@BeforeEach`: create `GetUserByIdUseCase`, assign both `InMemoryUserQueryRepository` and `InMemoryUserRepository` (for fallback)
- `InMemoryUserRepository` — implements `UserRepository` (existing interface): implement `findById(Long)` returning configurable `User`, other methods can throw `UnsupportedOperationException`
- Tests:
  1. `shouldReturnUserWithPackageStatus` — userQueryRepo has user → return it
  2. `shouldReturnNoPackageStatusWhenUserExistsButNoPackages` — userQueryRepo returns null, userRepo has user → return UserWithPackageStatus with NO_PACKAGE status
  3. `shouldThrowEntityNotFoundWhenUserDoesNotExist` — both repos return null → EntityNotFoundException
  4. `shouldThrowWhenUserIdIsNull` — BadRequestException

**IMPORTANT patterns:**
- Use `@BeforeEach` to create fresh instances (no shared state between tests)
- Use JUnit 5 assertions only: `assertEquals`, `assertNotNull`, `assertThrows`, `assertTrue`
- Test method names: `should[ExpectedBehavior]` or `should[ExpectedBehavior]When[Condition]`
- Each test class is self-contained — inner static classes for test doubles
- Domain objects use plain POJO pattern (setters), so no reflection needed for new models
- For existing `User.java` (Lombok @Builder), use `User.builder().name(...).build()` and reflection for id field if needed

**Tests for `getUsers` (GET /api/users):**
1. `shouldReturn200WithUsersFilteredByHq` — set headquartersId=1L, organizationId=null → assert response status 200, assert stub's execute was called (capture params)
2. `shouldReturn200WithUsersFilteredByOrg` — headquartersId=null, organizationId=1L → assert response status 200
3. `shouldReturn200WithAllUsers` — both null → assert response status 200, assert GetAllUsersUseCase was called
4. `shouldReturn400WhenBothHqAndOrgProvided` — headquartersId=1L AND organizationId=1L → assert response status 400
5. `shouldReturn400WhenUseCaseThrowsBadRequest` — configure stub to throw BadRequestException → assert 400
6. `shouldReturn500WhenUnexpectedError` — configure stub to throw RuntimeException → assert 500

**Tests for `getUserById` (GET /api/users/{id}):**
7. `shouldReturn200WithUserById` — configure stub to return user → assert 200
8. `shouldReturn404WhenUserNotFound` — configure stub to throw EntityNotFoundException → assert 404
9. `shouldReturn400WhenInvalidId` — configure stub to throw BadRequestException → assert 400

**Test approach:**
- Invoke `resource.getUsers(...)` or `resource.getUserById(...)` directly (no HTTP client — test the method, not the framework)
- Assert on `Response.getStatus()` for HTTP status codes
- Assert on `Response.getEntity()` to verify ApiResponse wrapping (cast to ApiResponse and check success/error)
- Do NOT test @Authenticated — that's a framework concern tested via E2E tests

**IMPORTANT:** The stubs extend concrete use case classes. Since use cases have `@Inject` fields, the stubs inherit those fields but they remain null — that's fine because the overridden `execute()` method never touches the repository. If the use case constructor does anything complex, the stub pattern still works because `@ApplicationScoped` CDI beans have no-arg constructors.
